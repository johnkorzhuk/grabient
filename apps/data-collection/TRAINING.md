# Training handoff: query → cosine palette model

Reference for whoever (or whatever agent) trains the model. Everything about
how the dataset was produced, what the files mean, how to evaluate, and the
sharp edges. The data-collection harness itself is documented in `README.md`;
this file is about *consuming* its output.

## 1. The task

Map a free-text search query (any language, any register, possibly emoji) to a
renderable palette:

- **Primary target**: 12 floats — cosine gradient coefficients, row order
  `[a.r, a.g, a.b, b.r, b.g, b.b, c.r, c.g, c.b, d.r, d.g, d.b]`, 3 decimals.
  Each RGB channel renders as `channel(t) = a + b·cos(2π(c·t + d))` for
  t ∈ [0,1], clamped to [0,1].
- **Secondary targets** (presentation): `style` (one of `linearGradient |
  linearSwatches | angularGradient | angularSwatches | radialGradient |
  radialSwatches`), `steps` (int 2–50, site default 7), `angle` (int 0–360,
  CSS convention). These describe how the palette renders best — they are
  a function of the palette, not the query.
- The task is **one-to-many in both directions**: a query has many valid
  palettes; a palette satisfies many queries. Train and evaluate accordingly
  (see §5).

Rendered swatch `i` of `steps` samples `t = i/(steps-1)`, both endpoints
included. Visual inspection of any palette: `https://grabient.com/<seed>`
(optionally `?style=&angle=&steps=`).

## 2. Data provenance (what biases to expect)

Generated by a closed-loop harness (Claude Sonnet generation, Claude Opus
judging) against a Cloudflare Worker that owns all state:

- **Forward**: invent queries targeting coverage gaps → author 2–4 candidate
  palettes each. Intent-first language.
- **Caption**: deterministic coefficient-space sampler creates palettes; the
  LLM writes 2–3 queries for each while looking at it. Description-leaning
  language. Caption pairs outnumber forward pairs roughly 4:1 (rebalancing to
  ~3:1 began 2026-07-18).
- **Judge**: scores each (query, palette) pair 0–10 blind against rendered
  PNGs. Verdicts: `ok | bad-match | bad-palette`. Calibration held mean
  ≈ 6.6 with zero drift across the whole run (audited).
- **Audit**: blind re-judging of random samples; promotes twice-confirmed
  pairs to the **golden** eval set; rejects slip-through bad palettes.
- **Triage** (added 2026-07-18): a panel of free Workers AI models
  pre-screens pending pairs; unanimous-"bad" pairs are auto-rejected
  (`judge_notes` prefixed `triage:`, status `rejected` — they never reach
  any export). Non-rejected pairs may carry `triage_votes` as metadata; it
  is not exported and the Opus judge treats it as a weak hint only.
- **Dedup**: three-tier (exact seed / Vectorize kNN over LAB features / exact
  avg-deltaE rescore, threshold 8). A Vectorize async-indexing race let ~2.5%
  near-dups in before 2026-07-18 ~12:30; those were swept (status
  `rejected`) and the race was closed. Pool verified dup-free at the <8
  avg-deltaE bar.

Known distribution characteristics (as of handoff):

- Registers: short/verbose/casual/typo spread is good; **1–2-word head terms
  and non-English are underrepresented (~2% each, steering to ~8%/~5% began
  2026-07-18 mid-day)**; emoji queries (styleHint `emoji`) began the same day.
- Coefficient space is well covered (amplitudes through the vivid range,
  frequencies 0.25–1.25, offsets centered). Very dark and very light palettes
  are thinner than mid-tones; high-contrast (0.25+ band) thinner still.
- `themes` exist for ~30% of palettes (caption-visited ones only).
- Queries never contain presentation language (enforced + retroactively
  purged) — presentation must be inferred from the palette, not the text.


### SFT rows (`format=sft`) — pairs with score ≥ 7, verdict ok

```json
{"query":"burnt orange autumn","coeffs":[0.5,...12 floats],"seed":"_gEh...",
 "style":"linearSwatches","steps":9,"angle":45,
 "score":8,"weight":1,"tags":["warm","vivid"],"category":"season-weather-time",
 "styleHint":"short","ambiguity":"medium","source":"forward","split":"train"}
```

- `weight`: SFT loss weight. 1.0 default; 3.0 for pairs the owner endorsed
  from the dashboard (`human_label='good'`). Human-endorsed pairs also enter
  at a lowered score threshold (≥5 instead of ≥7, judge verdict overridden) —
  heavy human weight with an absolute judge floor. Owner-vetoed pairs
  (`bad-match`) and pairs of owner-rejected palettes never appear.

- `coeffs` is the training target. Already **canonicalized**: globals
  normalized to defaults, rounded to 3 decimals, one consistent parameter
  regime (offsets/amplitudes practically within [0.2, 0.8]/|b|≲0.6, freq
  0.25–1.25). Do not re-normalize parameterization; do use the manifest
  stats (below) for numeric normalization.
- `style`/`steps`/`angle` are the palette's presentation (auxiliary heads).
- `score` (7–10 here) can weight the loss; `ambiguity` (`low|medium|high`,
  judge-emitted) is how constraining the query is — useful for loss
  weighting and eval slicing, sometimes null.
- `category`, `styleHint`, `source`, `tags` are slicing metadata — never
  model inputs.
- `split`: deterministic 90/5/5 (`train|val|test`) by hash of query id — all
  pairs of one query share a split; re-exports are stable.

The export manifest (in the Worker's `exports` table, `filters` column)
carries per-coefficient `mean/std/min/max` over the exported rows — use for
input/output normalization at training time.

### DPO rows (`format=dpo`) — preference pairs

```json
{"query":"...","chosen":[12 floats],"rejected":[12 floats],
 "chosenSeed":"...","rejectedSeed":"...","scoreGap":4,"split":"train"}
```

Best vs worst scored palette for queries where the gap ≥ 3. Count grows as
the judge drains the pending queue (most sibling candidates land there).
Optional preference stage after SFT.

### Eval rows (`format=eval`) — golden set, grouped per query

```json
{"query":"...","category":"...",
 "references":[{"coeffs":[...],"style":"...","steps":7,"angle":90,"score":9}, ...]}
```

Golden = independently confirmed by two blind reviews (original judge ≥7 ok
AND blind audit re-score ≥8). While goldens are scarce the export falls back
to top-scored test-split pairs (manifest `filters.source` says which you
got — check it).

## 4. Non-negotiable data rules

1. **Exclude golden queries from all training data.** The exporter now
   guarantees this server-side (golden queries — audit- or human-promoted —
   are filtered out of SFT and DPO; manifest reports
   `goldenQueriesExcluded`). Keep a training-time filter against the eval
   file as defense-in-depth anyway. The exam must stay unseen.
2. **Never train on `val`/`test` splits.** Use `val` for early stopping /
   hyperparameters; touch `test` once.
3. **Loss belongs in perceptual space, not raw coefficient space** (or at
   minimum: report perceptual metrics). Coefficient space is degenerate —
   visually near-identical gradients can have different coefficients (phase
   wraps, amplitude sign flips). The dataset's canonicalization removes most
   of this, but the honest distance between two palettes is: render both
   (8 samples each via the cosine formula), convert to LAB, average deltaE
   across positions, take min over the reversed orientation. That function is
   `comparePalettes`/`computeLabSamples` in
   `packages/data-ops/src/similarity` — reuse it, do not reimplement.
4. **Clamp/validate model outputs** to the data regime (offsets/amps sane,
   |freq| ≤ 2) before rendering; the site clamps channels to [0,1] at render.

## 5. Evaluation protocol

Primary metric — **nearest-reference perceptual distance** on the golden set:

```
for each eval query:
    d = min over references of avgDeltaE(model_palette, reference)   # both orientations
report: mean d; share of queries with d < 8 ("lands in a confirmed-good region");
        share with d < 16 ("plausible neighborhood")
```

Secondary:
- **Presentation accuracy**: style exact-match %, steps within ±2, angle
  within ±45° (circular), against the nearest reference.
- **Diversity** (the one-to-many check): sample k=5 outputs per query at
  whatever sampling temperature the model supports; mean pairwise deltaE
  among the 5 should be well above 8 for `high`-ambiguity queries. A model
  that collapses to one palette per query is failing half the task even if
  its nearest-reference distance is good.
- **Slices**: report every metric by `category`, `ambiguity`, `styleHint`
  (short vs verbose vs emoji), and language (non-ASCII heuristic). The
  aggregate hides exactly the registers users notice.

Baselines to beat before celebrating: (a) nearest-neighbor retrieval — embed
the query, return the nearest training palette (strong baseline given the
data volume); (b) per-category mean coefficients (weak floor).

## 7. Modeling notes (recommendations, not mandates)

- **Two viable shapes**: (a) fine-tune a small LM to emit the 12 floats (+
  style/steps/angle) as text — handles arbitrary query language including
  emoji for free, sampling gives diversity; (b) regression/diffusion head on
  a multilingual text-embedding backbone — cheaper to serve, but diversity
  and emoji/multilingual handling need explicit design. Given one-to-many
  structure, (a) with temperature sampling or (b) with a conditional
  diffusion/mixture head fit the task better than deterministic regression,
  which learns the *mean* of valid palettes — usually mud.
- Floats-as-text: emit 3-decimal fixed format exactly as in the data; the
  consistent regime + rounding exists so the vocabulary stays tiny.
- `score` can weight SFT loss (8–10 worth more than 7).
- Presentation heads are auxiliary — small loss weight; they must not fight
  the coefficient loss.
- Sanity harness: render model outputs for ~30 fixed prompts into a PNG grid
  after each run (reuse `apps/data-collection/harness/render.ts` patterns)
  and *look at them*. Metrics miss mud that eyes catch instantly.

## 8. Quick DB queries the researcher may want

```sh
# via: npx wrangler d1 execute grabient-dc --remote --json --command "..."
# (CLOUDFLARE_ACCOUNT_ID required; see §3)

-- SFT-eligible count
select count(*) from pairs p join palettes pl on pl.seed=p.palette_seed
 where p.status='scored' and p.verdict='ok' and p.score>=7 and pl.status!='rejected';

-- golden count / DPO-eligible queries
select count(*) from pairs where golden=1;
select count(*) from (select query_id from pairs where status='scored'
 group by query_id having count(*)>=2 and max(score)-min(score)>=3);

-- register shares
select count(*) from queries where length(text)-length(replace(text,' ',''))+1 <= 2;
select count(*) from queries where text glob '*[^ -~]*';
```
